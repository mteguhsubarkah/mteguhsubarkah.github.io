The Problem

KEMFOOD operates a franchise network of food shops. Each location needs to manage daily sales, track inventory in real time, order supplies from approved suppliers, and report to the head shop. Their workflow was entirely manual — WhatsApp messages for orders, spreadsheets for inventory, and no visibility into stock levels across locations. Double-ordering, lost orders, and payment reconciliation nightmares were daily occurrences.

The Solution

I designed and built a complete digital platform from scratch: a React Native mobile app for shop operations, backed by an Express.js REST API with PostgreSQL, covering the full business domain — POS transactions, procurement, inventory management, and BCA Virtual Account payment integration.

Every layer was designed with transactional consistency, idempotent writes, and concurrency-safe operations as first-class engineering concerns — not afterthoughts.

Production Engineering Concerns

The invisible work that separates production systems from toy projects

Race Condition Prevention

Critical

Multiple shop operators can update inventory or submit orders simultaneously. Without protection, concurrent writes cause lost updates — the classic read-modify-write race.

Scenario

Operator A reads inventory: stock = 50. Operator B reads same row: stock = 50. A adds 10 received goods → writes 60. B adds 5 → writes 55. Result: 5 units vanish. This is the classic lost update problem.

Solution

Used SELECT ... FOR UPDATE row-level locking on inventory reads within transactions. The second writer blocks until the first commits, then reads the updated value. Combined with optimistic locking via a version column for non-critical paths — if versions mismatch, the client retries. 

-- Atomic inventory update inside transaction
BEGIN;
  SELECT stock, version FROM inventory
    WHERE shop_name = 'shop-a'
    AND product_id = 'prod-123'
    FOR UPDATE;  -- row-level lock

  UPDATE inventory
    SET stock = stock + 10,
        version = version + 1,
        updated_at = NOW()
    WHERE shop_name = 'shop-a'
    AND product_id = 'prod-123';
COMMIT;
Zero lost updates in production — concurrent inventory operations are serialized at the row level

Idempotent Write Operations

Critical

Mobile networks are unreliable. A user taps "Submit Order," the request succeeds but the response times out. The app retries. Without idempotency, you get duplicate orders, double inventory deductions, or double payments.

Scenario

POST /send-inquiry creates an order and deducts stock. Network timeout → client retries → second order created with same items. Customer is charged twice. Inventory double-deducted.

Solution

Every mutating POST endpoint accepts an idempotency_key header (UUID generated client-side). The server stores completed keys in an idempotency_keys table with a unique constraint. On retry, the server returns the cached response instead of re-executing the operation. 

// Express middleware — idempotency guard
const idempotencyGuard = async (req, res, next) => {
  const key = req.headers['idempotency-key'];
  if (!key) return next();

  const existing = await db.query(
    'SELECT response_body, status_code FROM idempotency_keys WHERE key = \$1',
    [key]
  );

  if (existing.rows.length > 0) {
    // Return cached response — no side effects
    return res
      .status(existing.rows[0].status_code)
      .json(existing.rows[0].response_body);
  }
  next();
};
Safe retries — duplicate POST requests return cached results without re-executing business logic

Transactional Consistency

Critical

The receiving-buy-order endpoint must update the PO status to "received" and increment inventory atomically. If either fails independently, the system enters an inconsistent state — goods marked received but stock not updated, or vice versa.

Scenario

PO status updated to "received" → server crashes before inventory increment → PO shows received but stock is wrong. Manual reconciliation nightmare across multiple shops.

Solution

Wrapped cross-table mutations in PostgreSQL transactions with explicit BEGIN / COMMIT / ROLLBACK. If any step fails, the entire operation rolls back — the system never enters a half-updated state. Used the same pattern for order submission (cart → inquiry + stock reservation). 

// Receiving buy order — atomic cross-table update
const client = await pool.connect();
try {
  await client.query('BEGIN');

  // 1. Update PO status
  await client.query(
    'UPDATE buying SET status = \$1 WHERE po_number = \$2',
    ['received', poNumber]
  );

  // 2. Increment inventory (with row lock)
  for (const item of lineItems) {
    await client.query(
      'UPDATE inventory SET stock = stock + \$1 WHERE product_id = \$2 AND shop_name = \$3',
      [item.qty, item.product_id, shopName]
    );
  }

  await client.query('COMMIT');
} catch (err) {
  await client.query('ROLLBACK');
  throw err;
} finally {
  client.release();
}
All-or-nothing operations — PO status and inventory are always in sync

Schema Design & Indexing Strategy

Design

The database isn't just "tables that store data." It's the performance backbone and correctness enforcer. Every query pattern from the 30+ endpoints informed the schema design and index placement.

Design Decisions

3NF normalized — products, shops, suppliers, customers in separate tables with FK relationships. No data duplication.

Composite indexes on (shop_name, status, created_at) because the most frequent queries filter by shop + status + date range.

Partial indexes on active orders only — no point indexing completed/cancelled orders that are rarely queried.

UUID primary keys instead of auto-increment to prevent enumeration attacks and support future distributed sharding. 

-- Composite index for the most common query pattern
CREATE INDEX idx_inquiry_shop_status_date
  ON inquiries (shop_name, order_status, created_at DESC);

-- Partial index — only active orders (80% of queries)
CREATE INDEX idx_buying_active
  ON buying (shop_name, created_at DESC)
  WHERE status NOT IN ('done', 'cancelled');

-- Unique constraint for idempotency
CREATE UNIQUE INDEX idx_idempotency_key
  ON idempotency_keys (key);
Query patterns drove index design — not the other way around

Caching Strategy & Cache Invalidation

Optimization

Product catalogs and customer types are shared across all shops and change infrequently — but are queried on every transaction. Without caching, every POS operation hits the database for the same catalog data.

Cache Tiers

In-memory (app-level) — Product catalog and customer types cached with TTL of 5 minutes. Shared across requests via module-level Map.

Client-side — React Native caches catalog in AsyncStorage with ETag-based conditional requests. GET /get-product-byheadshop/:headShop returns 304 Not Modified if catalog hasn't changed.

No caching on — inventory, cart, and order status. These are high-write, high-read consistency paths that must always reflect the latest state.

Invalidation Strategy

TTL-based for simplicity — product catalog refreshes every 5 minutes. Acceptable staleness for a food franchise (products don't change mid-day). On POST /add-product, cache is explicitly invalidated for that head shop to ensure immediate visibility. This avoids the "cache invalidation is hard" problem by keeping the invalidation scope narrow. 

// Simple TTL cache for product catalog
const catalogCache = new Map();
const CACHE_TTL = 5 * 60 * 1000; // 5 minutes

async function getProductsByHeadShop(headShop) {
  const cached = catalogCache.get(headShop);
  if (cached && Date.now() - cached.timestamp < CACHE_TTL) {
    return cached.data; // cache hit
  }

  const result = await db.query(
    'SELECT * FROM products WHERE head_shop = \$1', [headShop]
  );

  catalogCache.set(headShop, {
    data: result.rows,
    timestamp: Date.now()
  });
  return result.rows;
}

// Explicit invalidation on product creation
async function addProduct(req, res) {
  // ... insert product ...
  catalogCache.delete(req.body.headShop); // invalidate
}
Cache what's read-heavy + rarely changes. Never cache what requires real-time consistency.

Error Handling & External API Resilience

Important

The BCA Virtual Account API is an external dependency. If it's down or slow, that shouldn't crash the entire POS system. Payment failures must be handled gracefully without leaving orders in broken states.

Scenario

Order submitted → BCA API timeout during VA generation → order stuck in limbo (submitted but no payment reference). User can't pay, can't resubmit, support ticket created.

Solution

Timeout + Retry with exponential backoff on BCA API calls (3 retries, 1s/2s/4s delays).

Circuit breaker pattern — after 5 consecutive failures, stop calling BCA for 30 seconds and return a "payment temporarily unavailable" response so orders can still be created (VA generated later via background job).

Decoupled order creation from payment — order is created first, VA generation is a separate async step. If VA fails, order exists but payment is retried.

External API failure never blocks core POS operations — graceful degradation by design

Security & Input Validation

Important

A POS system handles financial transactions and business data. SQL injection, unauthorized cross-tenant data access, and payment webhook spoofing are all real attack vectors.

Defense Layers

Parameterized queries everywhere — never string-concatenated SQL. All \$1, \$2 placeholders prevent SQL injection.

Tenant scoping enforced at query level — every query includes WHERE shop_name = \$1. Even if a user manipulates the request, they can only access their own shop's data.

BCA webhook signature verification — payment callbacks validated via HMAC-SHA256 signature before processing. Rejects spoofed payment confirmations.

Multer file validation — file size limits, content-type whitelist (image/* only), and filename sanitization to prevent path traversal attacks via crafted filenames. 

// Parameterized query — SQL injection proof
const result = await db.query(
  `SELECT * FROM inventory
   WHERE shop_name = \$1 AND product_id = \$2`,
  [shopName, productId] // never concatenated
);

// BCA webhook signature verification
const expectedSig = crypto
  .createHmac('sha256', BCA_SECRET)
  .update(rawBody)
  .digest('hex');

if (req.headers['x-bca-signature'] !== expectedSig) {
  return res.status(401).json({ error: 'Invalid signature' });
}
Defense in depth — parameterized queries, tenant isolation, webhook auth, file validation

Observability & Debugging in Production

Optimization

When a franchise owner says "my order from yesterday is missing," you need to trace exactly what happened. In a system with 6 modules, 30+ endpoints, and multiple state transitions, structured logging and request tracing are essential.

Observability Stack

Structured JSON logging on every request — method, path, shop_name, user, status code, response time. Enables grep-based debugging and log aggregation.

Request correlation ID — UUID propagated through the entire request lifecycle. When order creation fails, the correlation ID links the API log → DB transaction → BCA API call → error response.

Slow query logging — PostgreSQL log_min_duration_statement = 200ms flags queries that need index optimization. Found and fixed 3 missing indexes in the first month of production.

Health check endpoint — GET /health returns DB connection pool status, uptime, and last successful BCA API call timestamp. Used for monitoring alerts. 

// Request logging middleware with correlation ID
const requestLogger = (req, res, next) => {
  const correlationId = req.headers['x-correlation-id'] || uuidv4();
  req.correlationId = correlationId;
  res.setHeader('x-correlation-id', correlationId);

  const start = Date.now();
  res.on('finish', () => {
    console.log(JSON.stringify({
      timestamp: new Date().toISOString(),
      correlationId,
      method: req.method,
      path: req.path,
      shop: req.params.shopName || req.body?.shopName,
      statusCode: res.statusCode,
      durationMs: Date.now() - start
    }));
  });
  next();
};
Every production issue traced to root cause within minutes via correlation IDs and structured logs

API Design Patterns

Decisions that demonstrate production-grade backend thinking

Domain-Driven Route Organization

Each business domain has its own route file and controller. When the buying logic changes, you touch one module — not a 2000-line monolith router.

routes/buyingRoutes.js → controllers/buyingController.js
routes/cartRoutes.js → controllers/cartController.js
routes/inventoryRoutes.js → controllers/inventoryController.js
routes/productRoutes.js → controllers/productController.js
routes/supplierRoutes.js → controllers/supplierController.js
routes/customerRoutes.js → controllers/customerController.js

Proper HTTP Verb Semantics

Not just POST for everything. GET for retrieval, POST for creation, PUT for updates, DELETE for removal. Makes the API self-documenting and cacheable by proxy servers.

GET /cart/:shopName — read (cacheable)
POST /add-cart — create (idempotent w/ key)
PUT /update-cart — update (full replace)
DELETE /delete-cart-by-id/:id — remove

Explicit State Transition Endpoints

Order status changes use dedicated endpoints — not a generic "update status" PATCH. This enforces valid state transitions at the route level and creates natural audit boundaries.

POST /send-inquiry — Draft → Submitted
PUT /update-inq-process/:id — → Processing
PUT /update-inq-orderstatus/:id — → Sent
POST /update-buying-status-by-po — PO transitions
// Each endpoint validates the current state before transitioning

Business reporting needs drove specialized query endpoints with date-range filtering and status-specific views — real business intelligence, not just CRUD.

Multi-Tenant Scoping Pattern

Every data query includes tenant scope. Even if a request is crafted maliciously, the SQL WHERE clause limits visibility to the authenticated shop's data.

/inventories/:shopname — shop-level
/get-product-byheadshop/:headShop — franchise-level
/customers/:headshop — aggregated
// Scope enforced in SQL, not just route params
WHERE shop_name = \$1 — always parameterized

Challenges & Solutions

Real engineering problems solved in building a production POS

Challenge

Multi-Shop Data Isolation

Each shop needs its own inventory, orders, and suppliers — but the head shop needs aggregated views for catalog and customer management.

Solution

Dual-scope architecture: operational data scoped by shop_name, master data by head_shop. Enforced in SQL WHERE clauses, not just app logic.

Clean isolation with franchise-level aggregation

Challenge

Complex Order State Management

Orders traverse 6+ states across two different lifecycles (sales and procurement). Invalid transitions corrupt business data.

Solution

Dedicated state-transition endpoints that validate current state before transitioning. Each endpoint is a guard — you can't skip "processing" and jump to "sent."

Zero invalid state transitions in production

Challenge

BCA Payment API Reliability

External bank API goes down periodically. Payment failures can't block the entire POS workflow or leave orders in limbo.

Solution

Decoupled order creation from payment. Retry with exponential backoff, circuit breaker after 5 failures. Orders always created — VA generation retried asynchronously.

POS never blocked by bank API downtime

Challenge

Inventory-Procurement Sync

Goods receiving must update both PO status and inventory atomically. Partial failures leave data inconsistent.

Solution

PostgreSQL transaction wrapping both operations. If inventory update fails, PO status rolls back. Separate post-inventory vs add-inventory for initial stock vs incremental additions.

Procurement and inventory always in sync

Key Takeaways

Lessons from building a production business platform solo

01

Domain Modeling Before Code Saves Months

Understanding the franchise's actual workflow — how orders flow between shops and suppliers, how inventory is tracked, how payments are reconciled — meant the API mapped naturally to business processes. Zero rewrites needed.

02

State Machines Make Complex Flows Manageable

Making each state transition an explicit endpoint prevented invalid transitions and made the mobile app's UI logic straightforward — each screen maps to a state. Debugging is trivial: "which endpoint was last called?"

03

Multi-Tenancy Must Be Day-1 Architecture

Every query scoped by shop_name or head_shop was a Day 1 decision. Retrofitting tenant isolation into an existing system is a nightmare. The dual-scope pattern (shop-level ops, head-level master data) solved the franchise's organizational structure cleanly without a single data leak incident.

04

The "Invisible" Engineering Is What Keeps Production Alive

Idempotency keys, transaction boundaries, row-level locks, circuit breakers, structured logging — none of these show up in the UI. But they're the difference between a demo and a system that handles real money, real inventory, and real franchise owners who call you at midnight when something breaks.

05

Solo Full-Stack Forces Systems Thinking

When you own the database, the API, and the mobile app, you naturally design each layer to support the others. Every endpoint exists because a screen needs it. Every index exists because a query is slow. There's no "throw it over the wall" — you feel the pain of your own design decisions immediately.

06

External APIs Are Liabilities — Design Around Failure

The BCA Virtual Account API taught me that any external dependency will fail at the worst possible time. Decoupling order creation from payment, implementing retry with backoff, and adding a circuit breaker meant the POS kept working even when the bank didn't. Never let a third party's uptime define yours.
