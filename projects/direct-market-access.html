Project Overview

Direct Market Access (DMA) is a FIX Protocol-based trading gateway that I developed at Mirae Asset Sekuritas Indonesia, enabling institutional clients to execute stock orders directly on the Indonesia Stock Exchange (IDX). This was the first implementation of algorithmic and direct market trading at the company, opening an entirely new revenue channel for institutional business.

The system uses the Financial Information eXchange (FIX) protocol — the industry standard for electronic trading communication — to connect institutional trading terminals like Bloomberg EMSX and Marvelsoft directly to the exchange. The gateway handles order routing, execution reports, session management, and real-time monitoring through a custom-built admin console.

The core challenge was building a highly reliable, low-latency system that could handle the complexities of FIX message processing — including asynchronous execution reports, multi-threaded session management, queue-based message delivery, and automatic reconnection handling — all while maintaining strict compliance with exchange regulations.

Understanding FIX Protocol

The industry standard for electronic trading communication

The Financial Information eXchange (FIX) protocol is the global standard for electronic communication in financial markets. Used by exchanges, brokers, and institutional investors worldwide, FIX enables standardized order routing, execution, and post-trade communication.

Typical Order Flow

New Order Single (35=D)

Client sends order via terminal

Execution Report (35=8)

Gateway confirms order received (Ack)

Order Routed to Exchange

Gateway forwards to IDX via FIX 4.2

Execution Report (35=8)

Fill / Partial Fill / Reject from exchange

Execution Report to Client

Gateway relays execution status to terminal

Challenges & Solutions

Technical complexities of building a FIX trading gateway

Challenge

Asynchronous Execution Report Processing

Execution reports from the exchange arrive asynchronously and must be matched to the correct client order, processed in the right sequence, and delivered reliably — even during high-volume periods.

Solution

Implemented multi-threaded processing with dedicated task schedulers, blocking queues for ordered task execution, and scheduled reconciliation jobs that scan for unprocessed reports every 500ms.

Zero missed execution reports in production

Challenge

Multi-Terminal Order ID Mapping

Bloomberg and Marvelsoft use different order identifier formats. Mapping client order IDs to exchange order IDs across different terminal types while maintaining correct routing.

Solution

Built custom order ID mapping layer that translates between terminal-specific formats and internal identifiers. Used database-backed ID splitting and reconstruction for Marvelsoft's unique format.

Seamless multi-terminal order tracking

Challenge

Message Delivery Reliability

Network interruptions between gateway and exchange could cause messages to be lost. Every execution report must be delivered to the client terminal — failure means financial discrepancy.

Solution

Implemented file-based message persistence for unsent messages with automatic retry mechanism. Scheduled jobs continuously attempt to deliver queued messages until successful.

Guaranteed message delivery with persistence

Challenge

Stale Order & Rejection Handling

Orders that get rejected without acknowledgment, or acknowledged orders that later fail, require complex state management to avoid leaving orphan orders in the system.

Solution

Built automated reconciliation tasks that check acknowledgment status before sending rejections, clean up stale orders, and generate unique execution IDs to prevent conflicts.

Automated state reconciliation every 600ms

Key Takeaways

Lessons from building a production FIX trading gateway

01

FIX Protocol is Deceptively Complex

While the specification seems straightforward, real-world FIX implementations involve handling edge cases in session recovery, sequence number gaps, and vendor-specific message quirks that aren't in the standard.

02

Every Message Must Be Accounted For

In financial systems, a single lost execution report means money discrepancy. Building persistence, retry mechanisms, and reconciliation into the core architecture is non-negotiable.

03

Multi-threading Requires Careful Design

Concurrent FIX message processing with multiple sessions demands careful thread management. Blocking queues and task schedulers provided the ordered execution guarantees needed for financial correctness.

04

Monitoring is Essential for Production Trading

Having a real-time admin console was invaluable for operations. Being able to see session states, pending orders, and message queues at a glance enabled rapid issue identification.
