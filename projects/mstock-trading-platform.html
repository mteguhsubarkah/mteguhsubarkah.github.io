Project Overview

M-STOCK is the flagship online trading platform by Mirae Asset Sekuritas Indonesia, one of the top securities brokerage firms in the country. I was a core contributor to the complete revamp of the existing Neo HOTS application, modernizing the entire technology stack from monolithic to microservices architecture.

The platform enables retail and institutional investors to trade stocks on the Indonesia Stock Exchange (IDX) with real-time market data, advanced order management, live order book streaming, and comprehensive portfolio tracking. During my 3+ years on the project, I worked on everything from core trading APIs to real-time data streaming and regulatory reporting.

Beyond M-STOCK, I also contributed to several adjacent platforms during my tenure at Mirae Asset, including the NAVI Mutual Fund marketplace, Direct Market Access for institutional clients, internal reporting portal, and a whistleblower incident reporting system.

Challenges & Solutions

Technical obstacles overcome during development

Challenge

Real-time Order Book at Scale

Delivering live order book updates to thousands of concurrent users during peak trading hours without latency or dropped messages.

Solution

Implemented MQTT protocol with Redis Pub/Sub for efficient fan-out. Used connection pooling and optimized serialization to minimize latency.

Sub-100ms update latency

Challenge

Transactional Consistency

Ensuring order execution consistency across multiple microservices while maintaining high throughput during volatile market conditions.

Solution

Implemented idempotency keys for all order operations and optimistic locking to prevent duplicate submissions and race conditions.

Zero order duplication in production

Challenge

Market Data Caching

Balancing data freshness with database load when millions of queries hit stock information endpoints during active trading sessions.

Solution

Multi-tier caching with Redis using different TTLs for static vs dynamic data. Background jobs refresh cache before expiration to prevent stampede.

90% cache hit rate, 10x less DB load

Challenge

Polyglot Service Reliability

Managing 5+ microservices in Python, Go, and Java while ensuring high availability and consistent communication patterns.

Solution

Deployed Consul for service discovery and health checks. Standardized API contracts, logging formats, and implemented circuit breakers.

99.9% availability during market hours

Key Takeaways

Lessons from 3+ years building a production trading platform

01

Polyglot Architecture Needs Discipline

Using Python, Go, and Java together offers flexibility, but requires strict API contracts, standardized logging, and consistent deployment practices across all services.

02

Financial Systems Demand Idempotency

Every money-related operation must be idempotent. Network failures and retries are inevitable — the system must handle them gracefully without double-executing transactions.

03

Caching Strategy is Make-or-Break

Multi-tier caching with intelligent invalidation can transform query performance, but stale data in trading is dangerous. Finding the right TTL balance is an art.

04

Real-time Needs Different Thinking

Building real-time features requires thinking about message ordering, reconnection logic, and graceful degradation in ways that traditional request-response systems don't demand.
